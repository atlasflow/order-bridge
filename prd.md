# **1\. Overview**

The Atlas Core Order Bridge API defines the JSON format used to transmit one or more orders from any source system into Atlas Core for processing, stock reconciliation, and fulfilment.

Source systems include, but are not limited to:

* Atlas Cassa \- the multi-terminal ePOS application

* E-commerce websites (WooCommerce, custom storefronts)

* Third-party applications and marketplaces

* Manual or scheduled batch export tools

Core assigns its own internal UUID to each order after receipt. **Source systems do not need to pre-allocate Core IDs.**

## **1.1 Design Principles**

**Source-agnostic** — the envelope identifies the originating system; the schema does not assume any particular source application.

**Batch-capable** — a single payload may contain one or many orders, enabling both real-time and batch transfer modes.

**Decimal-safe** — all monetary values are transmitted as strings with fixed decimal precision to avoid floating-point rounding errors.

**Traceability-ready** — line items carry optional batch references and plant passport identifiers to support full stock traceability.

## **1.2 Transfer Triggers**

The trigger field in the envelope indicates why the transfer was initiated. Supported triggers are:

| Value | Description |
| :---- | :---- |
| **realtime** | Sent immediately on order completion at the source system. |
| **eod\_batch** | End-of-day batch containing all orders processed during a trading session. |
| **manual** | Manually initiated export by an operator or administrator. |
| **offline\_flush** | Orders accumulated during an offline period, sent on network reconnection. |
| **push** | Initiated by an external application or integration (e.g. a marketplace webhook). |

# **2\. Transfer Envelope**

Every payload — whether it contains a single order or a batch — is wrapped in a transfer envelope object. The envelope carries metadata about the transfer itself, separate from the orders it contains.

## **2.1 Envelope Fields**

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **schema\_version** | string | **Yes** | Version of this schema the payload conforms to. Must be "1.3.4" for this specification. |
| **generated\_at** | string (ISO 8601\) | **Yes** | UTC timestamp at which the payload was generated by the source system. Format: YYYY-MM-DDThh:mm:ssZ. |
| **trigger** | string (enum) | **Yes** | The event that caused this transfer. See Section 1.2 for permitted values. |
| **source** | object | **Yes** | Describes the originating system. See Section 2.2. |
| **transfer\_id** | string (uuid) | Conditional | Required for all inbound transfers. Omitted on Core exports. A UUID v4 generated by the source system uniquely identifying this transfer payload. Core uses this value, in combination with source.site\_id, to guarantee idempotency \- duplicate submissions with the same transfer\_id from the same site are rejected. Must be regenerated on genuine retries only if the payload has changed; identical retries must reuse the same transfer\_id. |
| **orders** | array | **Yes** | One or more order objects. May contain a single order or a batch. See Section 3\. |

## **2.2 Source Object**

The source object identifies the system and site from which the orders originate.

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **type** | string (enum) | **Yes** | The category of source system. Permitted values: “core", “cassa", "website", "app". |
| **site\_id** | string | **Yes** | A stable, machine-readable identifier for the originating site or branch (e.g. "site\_richmond"). |
| **site\_name** | string | **Yes** | Human-readable name of the site or application (e.g. "The Palm Centre"). |
| **operator\_id** | string | null | No | The ID of the staff member who initiated the transfer, where applicable. Null for automated or system-generated transfers. |

**Note:** For website or third-party application sources, site\_id should be a stable slug that uniquely identifies the integration (e.g. "web\_palmcentre\_uk"). This allows Core to route and attribute orders correctly across multiple channels.

# **3\. Order Object**

Each element of the orders array represents a single transaction. Three order statuses are supported: pos, order and refund.

## **3.1 Order Root Fields**

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **origin\_ref** | string | **Yes** | The order reference or identifier as known by the originating system (e.g. "CASSA-2026-00412" or "WEB-2026-00103"). Core stores this for cross-reference but assigns its own UUID internally. |
| **status** | string (enum) | **Yes** | The state of the order. Permitted values: “pos”, "order", "refund". See Section 3.2. |
| **channel** | string | **Yes** | The sales channel through which this order was placed. Examples: "in-store", "website", "phone", "marketplace". |
| **operator\_id** | string | null | No | ID of the staff member who processed the order. Null for unattended or automated channels such as website checkouts. |
| **ordered\_at** | string (ISO 8601\) | **Yes** | UTC timestamp of when the order was created or completed at the source. Format: YYYY-MM-DDThh:mm:ssZ. |
| **notes** | string | null | No | Free-text order-level notes from the operator or customer. Stored against the order in Core. |
| **customer** | object | **Yes** | Identifies or describes the customer. See Section 3.3. |
| **fulfilment** | object | **Yes** | Describes how the order is to be fulfilled. See Section 3.4. |
| **items** | array | **Yes** | One or more line item objects. See Section 3.5. |
| **ancillaries** | array | null | **No** | One or more ancillary fee objects associated with the order. See Section 3.8. |
| **totals** | object | **Yes** | Order-level monetary summary. See Section 3.6. |
| **payments** | array | **Yes** | One or more payment records. See Section 3.7. |

## **3.2 Order Status Values**

| Value | Description |
| :---- | :---- |
| **pos** | A successfully completed sale. Items have been sold and payment received. No further action is required. |
| **order** | A successfully completed sale, which needs dispatching at a later date.  |
| **refund** | A full or partial refund against a previously completed order held in Core. The totals and payment amounts stay positive. The totals.refund\_of field must reference the Core UUID of the original order. |

**3.3 Customer Object**  
The customer object supports three distinct customer types, controlled by the required type field. Only the fields relevant to the declared type should be populated; irrelevant fields should be omitted or set to null.

### **3.3.1 Customer Type Values**

| Value | Description |
| :---- | :---- |
| **anonymous** | A walk-in customer with no identifying information. No further customer fields are required. |
| **registered** | An existing customer is already held in Core. Only the id field is required; Core will resolve the full customer record. |
| **new** | A new customer whose details should be created in Core upon receipt. Full name, contact, and address fields are expected. |

### **3.3.2 Customer Fields**

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **type** | string (enum) | **Yes** | Customer classification. See 3.3.1. |
| **id** | string | null | Conditional | Core customer ID. Required when type is "registered". Omit or null otherwise. |
| **name** | string | null | Conditional | Full name of the customer. Required when type is "new". This may be different from “contact”, (i.e. customer is a company and it has a designated contact person). |
| **email** | string | null | Conditional | Email address. Required when type is "new" if available. |
| **phone** | string | null | No | Phone number in E.164 format (e.g. "+447712345678"). Optional for new. |
| **address** | object | null | Conditional | Customer address. Required for new when fulfilment type is "delivery". See address structure in Section 3.4.2. |
| **contact** | string | null | Conditional | Customer’s designated contact name for the order. Required when the type is “new”, if available. |

## **3.4 Fulfilment Object**

The fulfilment object describes how and where the order will be delivered or collected. A single order supports both delivery and collection — the type field controls which additional fields are relevant.

### **3.4.1 Fulfilment Fields**

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **type** | string (enum) | **Yes** | Fulfilment method. Permitted values: "collection", "delivery". |
| **delivery\_slot** | object | null | Conditional | A delivery slot booking. Optional, and used only when the type is set to “delivery”. See Section 3.4.3. |
| **delivery\_address** | object | null | Conditional | Full delivery address. Required when type is "delivery". See Section 3.4.2. |
| **notes** | string | null | No | Free-text fulfilment notes (e.g. delivery instructions, access codes). |

### **3.4.2 Address Structure**

Used in both fulfilment.delivery\_address and customer.address.

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **line1** | string | **Yes** | First line of the address (house number and street). |
| **line2** | string | null | No | Second address line (flat number, building name, etc.). |
| **line3** | string | null | No | Third line of the address. |
| **city** | string | **Yes** | City or town. |
| **postcode** | string | **Yes** | Postal or ZIP code. |
| **region** | string | null | No | Administrative division (region, county, state, province etc.). Optional, omit or set to null where not applicable. |
| **country** | string | **Yes** | ISO 3166-1 alpha-2 country code (e.g. "GB", "IE", "IT"). |

### **3.4.3 Delivery Slot**

Optional object, used when fulfilment.type is set to “delivery”. 

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **delivery\_on** | string (ISO 8601\) | **Yes** | Delivery date. Informational only; the authoritative delivery date is held in Core. |
| **booking\_id** | string (uuid) | Conditional | Reference to a pre-booked delivery slot record in Core. If not provided, the Core will assign one automatically, based on its internal rules.  |
| **notes** | string | null | No | Notes related to the delivery slot. |

## **3.5 Line Item Object**

Each object in the items array represents a single product line within the order. VAT is declared per line, enabling mixed-rate orders (e.g. zero-rated plants alongside standard-rated sundries).

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **sku** | string | **Yes** | The product's stock-keeping unit code (variant) as held in Atlas Core or the source system's product catalogue. |
| **name** | string | **Yes** | The product name at the time of sale. This is a snapshot of the name and is stored verbatim — Core does not attempt to reconcile it against the live product name. |
| **qty** | number | **Yes** | Quantity sold. Must be a positive integer for completed orders; positive or fractional for items sold by weight or volume. |
| **uom** | string | **Yes** | Unit of measure for the quantity. Examples: "item", "kg", "litre", "bundle". |
| **unit\_price** | string (decimal) | **Yes** | Unit selling price excluding VAT, as a string with 4 decimal places (e.g. "42.6100"). Always positive. |
| **discount** | string (decimal) | **Yes** | Percentage discount applied to this line, as a string with 6 decimal places (e.g. "10.000000"). Use "0.000000" when no discount applies. |
| **vat\_rate** | string (decimal) | **Yes** | VAT rate applicable to this line as a percentage string with 2 decimal places (e.g. "20.00", "0.00"). |
| **batch** | string | null | No | Stock batch or lot reference for traceability. Should match the batch identifier used in Atlas Core's stock management. Null if not applicable or if the Core should allocate according to its own defined strategies and rules. |
| **passport** | string | null | No | Plant passport identifier, where required by plant health legislation. Null for non-regulated items. Note: in Italy, plant passports must be listed on the invoice. |
| **notes** | string | null | No | Free-text notes specific to this line item (e.g. condition notes, special handling instructions). |
| **line\_ex\_vat** | string (decimal) | Yes | Net line total.\= qty \* unit\_price \* (100 \- discount)/100 |
| **line\_vat** | string (decimal) | Yes | VAT amount for the line.\= qty \* unit\_price \* (100 \- discount)/100 \* vat\_rate/100 |

**Note:** All decimal string fields use consistent precision: 4 decimal places for monetary values (prices, totals, VAT), 6 decimal places for discount percentages. This matches Atlas Core's internal pricing precision and avoids floating-point drift across integrations.

## **3.6 Totals Object**

The totals object provides an order-level monetary summary. Core will validate these figures against the sum of the line items. Any discrepancy will cause the order to be rejected.

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **items\_net** | string (decimal) | **Yes** | Sum of all item lines, exclusive of tax, with 4 decimal places. Positive number. \=qty \* unit\_price \* (1 \- discount/100)  |
| **ancillaries\_net** | string (decimal) | **Yes** | Sum of all ancillary charges, exclusive of tax, with 4 decimal places. Zero if no ancillary items are present (e.g. “0.0000”) |
| **total\_vat** | string (decimal) | **Yes** | Sum of all vat values across all lines and ancillary items, with 4 decimal places.  |
| **grand\_total** | string (decimal) | **Yes** | Total order value including VAT: items\_net \+ ancillaries\_net+ total\_vat, with 4 decimal places. |
| **refund\_of** | string | null | Conditional | The Core-assigned signature (uuid/hash) of the original completed order being refunded. Required when status is "refund". Must be null for completed orders. |

## **3.7 Payment Object**

The payments array supports split and mixed payments. The sum of all payment amounts must equal the order's grand\_total. Each element in the array represents a distinct payment instrument used in the transaction.

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **method** | string (enum) | **Yes** | Payment method used. See Section 3.7.1 for permitted values. |
| **amount** | string (decimal) | **Yes** | Amount tendered via this method, with 4 decimal places. For split payments, each entry carries only its portion. |
| **reference** | string | null | No | Payment gateway or terminal reference for this transaction (e.g. Stripe charge ID, card terminal authorisation code). Null for cash payments. |
| **date** | string (ISO 8601\) | **Yes** | Payment date.  |

### **3.7.1 Payment Method Values**

| Value | Description |
| :---- | :---- |
| **cash** | Physical cash payment made at a till or in person. |
| **card** | Payment by debit or credit card via a physical terminal. |
| **account** | Charged to a trade account. The customer must be of type "registered" with a trade account held in Core. |
| **online** | Payment processed via an online payment gateway (e.g. Stripe, PayPal, Klarna). |
| **gift\_card** | Payment made using a gift card or voucher. The reference field should carry the gift card number or voucher code. |

## **3.8 Ancillary Object**

Items that are not SKUs and, for accounting purposes, must be held separately. These include (but are not limited to): delivery charges, custom charges, packaging, deposits etc. The categories are pre-defined in the Core. If needed, a separate API is available. 

| Field | Type | Required | Description |
| :---- | :---- | :---- | :---- |
| **category** | string | **Yes** | Category of fee. Examples: “delivery\_fee”, “phytosanitary\_certificate” etc. The actual format (free text or slug or number) is defined by the Core.  |
| **qty** | number | **Yes** | Quantity of the ancillary item (e.g. 1 for a flat fee). Must be a positive number |
| **unit\_price** | string (decimal) | **Yes** | Unit selling price excluding VAT as a string with 4 decimal places. Always positive.  |
| **vat\_rate** | string (decimal) | **Yes** | VAT rate applicable for this fee as a percentage string with two decimals (e.g., “20.00” or “0.00”) |
| **description** | string | null | No | A description of the ancillary item.  (e.g. “Delivery fees” or “Custom charges for paperwork”) |
| **total\_ex\_vat** | string (decimal) | Yes | Net line total.\= qty \* unit\_price |
| **total\_vat** | string (decimal) | Yes | VAT amount for the line.\= qty \* unit\_price \* vat\_rate/100 |

# **5\. Validation Rules**

Core applies the following validation checks on receipt of a transfer payload. Payloads that fail validation are rejected in their entirety; no partial order ingestion occurs.

## **5.1 Envelope Validation**

* schema\_version must be "1.3.3".

* generated\_at must be a valid ISO 8601 UTC timestamp, current or in the past

* orders must be a non-empty array

* When present, `transfer_id` must be a valid UUID v4. Core will reject any inbound transfer where the combination of `transfer_id` and `source.site_id` matches a previously accepted payload. Duplicate submissions carrying the same `transfer_id` from the same site are silently acknowledged without reprocessing. Identical retries must reuse the same `transfer_id`; a new `transfer_id` is only generated when the payload content has genuinely changed.

## **5.2 Order Validation**

* items must contain at least one line item.

* payments must contain at least one payment entry.

* The sum of payments\[\].amount must equal totals.grand\_total

* For refund orders, totals.refund\_of must be a valid Core order signature. Core will verify the referenced order exists and is in a refundable state.

* For refund orders, all monetary amounts (grand\_total, payment amounts) must be positive.

* customer.type "account" payments require customer.type to be "registered" and the customer must have this setting active. 

## **5.3 Line Item Validation**

Core validates the arithmetic of each line item and ancillary individually. Any line where the submitted values do not agree with the expected formulas, within the permitted tolerance, will cause the entire payload to be rejected.

The following rules apply to each object in the `items` array:

* All decimal string fields must be parseable as numeric values.  
* `line_ex_vat` must equal `qty × unit_price × (100 − discount) / 100`, within a tolerance of `0.0001`.  
* `line_vat` must equal `line_ex_vat × vat_rate / 100`, within a tolerance of `0.0001`.  
* The sum of all `line_ex_vat` values across all items must equal `totals.items_net`, within a tolerance of `0.0001`.  
* The sum of all `line_vat` values across all items and all ancillary `total_vat` values must equal `totals.total_vat`, within a tolerance of `0.0001`.

The following rules apply to each object in the `ancillaries` array, when present:

* `total_ex_vat` must equal `qty × unit_price`, within a tolerance of `0.0001`.  
* `total_vat` must equal `total_ex_vat × vat_rate / 100`, within a tolerance of `0.0001`.  
* The sum of all `total_ex_vat` values across all ancillary items must equal `totals.ancillaries_net`, within a tolerance of `0.0001`.

The following rule applies to the `totals` object:

* `grand_total` must equal `items_net + ancillaries_net + total_vat`, within a tolerance of `0.0001`.

## **5.4 Notes on idempotency**

The Order Bridge API is designed to be safe to retry. Source systems should feel confident resending a payload without risk of duplicate order creation in Core.

When a `transfer_id` is present, Core stores a composite key of `transfer_id` \+ `source.site_id` on successful ingestion. Any subsequent inbound transfer carrying the same composite key is recognised as a duplicate and silently acknowledged — Core returns a success response without reprocessing the payload. This ensures that transient network failures and retry attempts do not result in duplicate orders.

The following rules apply:

* When present, `transfer_id` must be a valid UUID v4.  
* `transfer_id` uniqueness is scoped to `source.site_id` — two different sites may generate the same UUID without conflict.  
* On a genuine retry of an unchanged payload, the sender must reuse the same `transfer_id`. Reusing the same ID is what guarantees safe deduplication.  
* A new `transfer_id` must only be generated when the payload content has genuinely changed and is intended to be processed as a new transfer.  
* `transfer_id` is required for all inbound transfers. It is omitted on payloads exported by Core.
